## 1) 제품 및 서비스의 목적·기능·특징

### 목적 및 배경

전국 사업장의 임금체불 건수는 매년 2만 건 이상(2023년 기준 2.3만 건, 피해액 1.7조 원)이지만, 근로감독관 1인당 담당 사업장은 수천 곳에 달합니다. 현행 체계는 **"신고가 접수된 뒤 조사"**하는 사후 적발 방식이기 때문에, 체불이 발생해도 피해 근로자가 신고하지 않으면 묻힙니다.

특히 장애인 근로자는 고용 불안·정보 접근성 문제로 체불 발생 시에도 신고율이 현저히 낮습니다.

**WageGuard**는 8개 공공기관 데이터를 교차 분석하여 체불 가능성이 높은 사업장을 사전에 AI가 자동 정렬 → 근로감독관·KEAD 담당자가 **"어디부터 볼지"** 를 즉시 판단할 수 있도록 지원합니다.

### 핵심 기능

| 기능 | 설명 |
|------|------|
| **점검 우선순위 정렬** | 전국 사업장을 8기관 교차 의심도(0~100점)로 자동 정렬. 고위험(65+)·주의(40~64)·최저임금위반 자동 분류 |
| **LIVE 7단계 실시간 검증** | 사업장명 입력 → 체불명단·국세청·네이버·Google·NPS·DART·AI종합 7개 API를 SSE 스트리밍으로 동시 호출 |
| **장애인 근로자 우선 보호** | KEAD 의무고용율 교차 추정으로 장애인 활동 사업장 위험점수 25% 가산. KEAD 3개 + 고용정보원 4개 데이터셋 결합 |
| **선행지표 3종** | DART 재무위험(부채비율·영업손실·자본잠식) · NPS 가입자 이탈 Z-score · 4대보험 삼각검증으로 체불 발생 **3~6개월 전** 탐지 |
| **부정수급 차단 SDK** | 브라우저 9개 신호(timezone·언어·WebRTC·WebGL·마우스·키보드 등)로 RDP 원격 접속·대리 수강 실시간 탐지 |
| **익명 체불 신고·공개 집계** | 근로자 익명 신고 접수 → 신뢰도 자동 산출(신고자 수×30점 + 다일자×15점 + 건수×5점) → 60점 이상 사업장 공개 |

### 시스템 구성

```
FastAPI + SQLite + Jinja2 + Tailwind CSS + ECharts + htmx
├── Track A: 부정수급 차단 SDK (wageguard-sdk.js)
├── Track B: 임금체불 위험 점검 우선순위 ML
└── 실시간 검증 파이프라인 (SSE EventSource)
```

## 2) 고용노동 데이터 활용 방안 (활용성)

### 활용 데이터 현황 — 8개 공공기관

| 제공기관 | 데이터명 | 활용 방식 |
|----------|----------|-----------|
| **고용노동부** | 체불사업주 명단 공개 | 789건 실데이터 직접 수집·DB 적재. ML 학습 라벨(체불=1) 및 의심도 기준점으로 활용 |
| **한국장애인고용공단(KEAD)** | 근로지원인 구인정보(15149876) · 수행기관 정보(15131282) · 고용개발원 보고서(15144216) | ML 9특성 중 2개를 KEAD 의무고용율(3.1%) 교차 추정으로 결합. Ablation 분석 결과 F1 +20.4%p 기여 확인 |
| **한국고용정보원** | 워크넷 직업정보·직무(NCS)·훈련과정·취업역량 강화 API (4종) | TF-IDF 임베딩 기반 직무-장애유형 매칭 유사도 산출. 장애인 고용 적합성 점수 생성 |
| **국민연금공단** | 사업장 가입자 현황 (시계열) | 월별 가입자 수 변동 Z-score 이상탐지. 이탈률 급증 → 임금체불 선행지표로 활용. 20,000개 사업장 커버 |
| **금융감독원(DART)** | 재무제표 공시 (전자공시 OpenAPI) | 부채비율·영업손실·자본잠식 3개 지표 조합 위험 점수화. 546건 재무위험 법인 탐지 |
| **국민건강보험공단** | 직장가입자 현황 | NPS·고용보험과 3-way 교차검증. 가입자 수 불일치 → 유령사업장·이중계약 탐지 |
| **근로복지공단** | 고용보험·산재보험 | 가입자 현황 삼각검증 4대보험 중 2개 |
| **국세청** | 사업자 상태 조회 (NTS) | 폐업·휴업·유효 상태 실시간 확인. LIVE 검증 7단계 중 2번째 단계 |

### 데이터 획득·지속성·가공

- **공개 API 방식**: 고용노동부 체불명단·국세청·DART는 공개 API 또는 웹 스크래핑으로 주기적 수집
- **KEAD·고용정보원**: data.go.kr 활용신청 완료(2026-05-05), OpenAPI 키 발급 후 정기 호출
- **NPS·건보**: 샘플 데이터 기반 시뮬레이션 → 실 연동 시 동일 파이프라인 적용 가능
- **가공 파이프라인**: `scripts/` 디렉터리 내 수집·정제·DB 적재 자동화 스크립트 완비

## 3) AI 활용 방안 (활용성)

### Track B — 임금체불 위험 분류 모델

| 항목 | 내용 |
|------|------|
| 알고리즘 | Logistic Regression (pure Python, scikit-learn) |
| 특성 수 | 9개 (체불이력·NPS이탈·DART위험·최저임금·폐업상태·KEAD교차 2개 등) |
| 학습 데이터 | 고용노동부 체불명단 789건 + 정상 사업장 시뮬레이션 샘플 |
| K-fold CV F1 | **0.928** (5-fold, PYTHONHASHSEED=0 고정) |
| Holdout F1 | **0.919** (80/20 분리) |
| KEAD 결합 효과 | Ablation 분석: KEAD 2특성 추가 → F1 **+20.4%p** |
| 해석 가능성 | 특성별 가중치·Confusion Matrix 실시간 계산 → `/ml` 페이지 라이브 표시 |

### Track A — 부정수급 실시간 차단 SDK

| 항목 | 내용 |
|------|------|
| 탐지 대상 | 고용장려금·직업훈련 보조금 부정수급을 위한 RDP 원격 접속, 대리 수강 |
| 신호 9종 | Timezone, 언어 설정, WebRTC IP, WebGL fingerprint, Canvas fingerprint, 마우스 패턴, 키보드 딜레이, 스크롤 패턴, 배터리 상태 |
| 판정 방식 | 신호별 가중 합산(레이어 A·B·C 3단계) → 임계치 초과 시 신청 차단 |
| 성능 | F1 **0.864** (1,000건 시뮬레이션) |
| 통합 방식 | `<script src="wageguard-sdk.js">` 한 줄 삽입으로 기존 시스템에 즉시 적용 |

### 추가 AI 기술

- **TF-IDF + 코사인 유사도**: 직무 공고 텍스트 ↔ 장애 유형 매핑 (워크넷 NCS 데이터)
- **Z-score 이상탐지**: NPS 가입자 월별 변동 시계열에서 이탈 이상 구간 자동 경보
- **몬테카를로 시뮬레이션**: 시스템 도입 효과(사회적 임팩트) 1,000회 반복 추정 → 평균 5,641억 원/년 체불 예방 효과 산출

### AI 지속성

- Logistic Regression은 신규 체불명단 수집 즉시 재학습 가능 (경량, 의존성 없음)
- 특성 중요도·Ablation 결과를 웹 UI에서 실시간 확인 → 모델 드리프트 모니터링

## 5) 차별성 (차별성)

### 포지셔닝

기존 시장에는 **매칭(워크넷)**과 **사후 적발(근로감독 행정)** 만 존재합니다.  
WageGuard는 그 사이의 **"사전 우선순위 정렬"** 빈 포지션을 점유합니다.

### 경쟁 서비스 비교

| 항목 | 기존 임금체불 신고 앱 | 고용노동부 체불명단 공개 | **WageGuard** |
|------|----------------------|------------------------|---------------|
| 탐지 시점 | 체불 발생 **후** 신고 | 체불 확정 **후** 공개 | 체불 발생 **3~6개월 전** 선행 탐지 |
| 데이터 소스 | 신고자 정보만 | 1개 기관 | 8개 공공기관 교차 |
| AI 분류 | 없음 | 없음 | Logistic Regression F1 0.928 |
| 장애인 특화 | 없음 | 없음 | KEAD 결합 → F1 +20.4%p |
| 부정수급 탐지 | 없음 | 없음 | SDK F1 0.864 |
| 실시간 검증 | 없음 | 없음 | 7개 API SSE 스트리밍 |
| 점검관 우선순위 | 없음 | 수동 검색 | AI 자동 정렬 |

### 핵심 차별점 3가지

1. **사전 탐지**: 체불명단 등재 전 재무·NPS·보험 신호로 3~6개월 앞서 포착
2. **장애인 근로자 특화**: 국내 최초 KEAD 데이터를 ML 특성에 직접 결합
3. **Track A + B 이중 방어**: 체불 탐지(Track B)와 부정수급 차단(Track A)을 단일 시스템에서 동시 제공

## 6) 사업화 계획 (효과성)

### 사회적 효과

- 임금체불 피해 근로자 **연 25만 명** 중 조기 발굴 목표
- 몬테카를로 시뮬레이션 1,000회: 시스템 도입 시 **연간 5,641억 원** 체불 예방 기대효과
- 장애인 근로자 체불 조기 발굴 → 고용 유지율 향상

### 홍보·마케팅

- 공모전 수상 후 고용노동부 보도자료·SNS 채널 통해 노출
- KEAD·한국고용정보원 공동 보도자료 배포
- 고용노동청 내부 워크숍 → 현장 감독관 대상 시연

### 실행 가능성 근거

- **현재 작동하는 시스템**: 로컬 http://127.0.0.1:8123 에서 전 기능 즉시 시연 가능
- **50+ 페이지, 200+ API 라우트** 완비 — 이미 제품 수준으로 구현됨
- **경량 아키텍처**: FastAPI + SQLite → 단일 서버 월 2~5만 원 수준 운영 비용
- **오픈소스 의존성만 사용**: 라이선스 리스크 없음, 즉시 배포 가능

### 확산 가능성

동일한 우선순위 정렬 메커니즘을 다음 5개 영역에 **코드 수정 없이** 이식 가능:

| 영역 | 운영주체 | 데이터 교체만으로 적용 |
|------|----------|----------------------|
| 외국인 근로자 체불 | 고용노동부 외국인력과 | 외국인 고용허가 + NPS 교차 |
| 청년 고용지원금 부정수급 | 고용노동부 청년고용과 | Track A SDK 직접 적용 |
| 고령자 고용장려금 | KEAD | 동일 SDK |
| 산재보험 부정청구 | 근로복지공단 | DART·NPS 교체 |
| 공공조달 계약 불이행 | 조달청 | 재무 데이터 교체 |
